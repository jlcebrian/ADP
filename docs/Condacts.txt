  Undocumented or weird DAAD Condacts
───────────────────────────────────────────────────────────────────────────────


INK	    n
PAPER	n

    Both condact use a look-up table which is platform-dependant in order
    to choose the actual color that will be used for the text:

            Color       ST/AMIGA    CGA         EGA/VGA

            0           0           0           0
            1           15          3           15
            2           1           2           4
            3           2           1           2
            4           3           0           1
            5           4           3           3
            6           5           2           5
            7           6           1           6
            8           7           0           7
            9           8           3           8
            10          9           2           9
            11          10          1           10
            12          11          0           11
            13          12          3           12
            14          13          2           13
            15          14          1           14

    The MS-DOS interpreter uses the parameter as-is in text mode.

    The 8 bit interpreters use the same logical color scheme, but the
    look-up table is not hardcoded: it is stored in the game's graphics
    database (the footer word ADP calls "coltab"). The table starts with
    a one byte prefix (a global bright flag on Spectrum, the border
    color on MSX and C64) followed by the conversion entries:

            Machine     Entries   Notes

            Spectrum    8         Bit 3 of the parameter (bright) is
                                  preserved and OR-ed over the result
            MSX         16        TMS9918 colors; all games published
                                  by Aventuras AD share the same table
                                  (1 maps to 14, gray)
            C64         16        CBM hardware colors

    The CPC interpreter has no table at all: INK is masked to the pen
    range (AND 3) and passed straight to the firmware, and the PCW
    interpreter ignores INK entirely.



CALL    address                             (Condact 0x65)

    Transfers execution to a machine code routine at the given address,
    used with code embedded in the database. The parameter is a single
    16 bit little-endian address (0-65535) — the only condact with a
    word-sized parameter, so it occupies 3 bytes in the process stream.
    Inline data bytes may follow the condact as static parameters for
    the routine.

    ADP treats CALL as a no-op: the only known use in a published game
    is Aventura Espacial part 1 (C64), where process 6 calls a 5
    instruction routine embedded in the database that enables the
    machine's keyboard auto-repeat (a purely local input tweak with no
    game logic effect).



MOUSE   flagno  op                  (Condact 0x56, replaces version 1 PROMPT)

    This condact requires two parameters. The second parameter is a
    command/operation number, similar to how GFX/SFX works:

            Command

            0       Reset/initialize the mouse
            1       Show the mouse cursor
            2       Hide the mouse cursor
            3       Get the mouse position & state
    
    The second parameter marks the first one of a table of four flags
    which are populated when command is 3 with the following info:

            Flag    Description

            n+0     Mouse button state
            n+1     Mouse X position, in 8x8 cell coordinates
            n+2     Mouse Y position, in 8x8 cell coordinates (row)
            n+3     Mouse X position, in 6x8 character coordinates

    The button state is a combination of the following values:

            Value

            1       Left mouse button is down
            2       Right mouse button is down
            4       Middle mouse button is down

SFX     value  op                   (Condact 0x12, two parameters)

    Direct sound-chip access. The interpretation of the second
    parameter is platform dependent; the first parameter is the value
    to apply.

    8-bit machines (Spectrum 128K/+2/+3, CPC, MSX, C64): the second
    parameter is a sound chip REGISTER number and the condact writes
    the first parameter into it, immediately and with no queuing. The
    game itself is responsible for silencing the chip afterwards.

            Machine     Chip        Registers   Access path

            Spectrum    AY-3-8912   0-13        ports 0xFFFD/0xBFFD
            CPC         AY-3-8912   0-13        firmware SOUND REGISTER
            MSX         AY-3-8910   0-13        BIOS WRTPSG
            C64         SID         0-24        direct store, unchecked

    Register numbers outside the valid range are ignored (except on
    C64, where the write happens regardless). On 48K Spectrums, which
    have no AY chip, the condact is a no-op: the interpreter only
    enables the sound patch vector when it detects a 128K machine.
    The Plus/4 interpreter stubs it out entirely.

    16-bit machines (verified in the Atari ST interpreter): a second
    parameter of 255 plays the sample currently held in the picture
    buffer, if the buffered resource is a sample (the first parameter
    is ignored). Any other second parameter uses the low two bits of
    the FIRST parameter to configure the console: bit 0 enables the
    keyboard click, bit 1 enables key auto-repeat.

    Example: Aventura Espacial part 3 produces its "white noise"
    portrait transition on the AY machines with this sequence (while
    rapidly printing random-pattern messages 157-161 to the portrait
    window for the visual counterpart):

            SFX 15 6        ; R6  noise period = 15
            SFX 15 8        ; R8  channel A volume = 15
            SFX 247 7       ; R7  mixer = %11110111, noise only on A
            ... visual noise loop ...
            SFX 0 6         ; noise period 0
            SFX 0 8         ; volume 0
            SFX 255 7       ; mixer: everything off

    The audible duration is implicit: it lasts as long as the game
    logic between the ON and OFF writes takes to run.

    The C64 version of the same game writes SID registers instead
    (attack/decay, frequency, control and the master volume register
    24), one register per SFX call, mirroring the AY sequences.

    ADP status: the 16-bit forms (sample playback and key click) are
    implemented. The 8-bit register writes are currently ignored.
    ADP already contains an AY register-level synthesizer used for
    the Atari ST EXTERN PSG streams (DDB_PSGState, DDB_RenderPSGTicks
    in ddb_psg.cpp, 50Hz ticks at 12.5kHz), so Spectrum/CPC/MSX SFX
    support only needs a persistent live PSG state fed by CONDACT_SFX
    register writes and streamed continuously through the audio
    backend, instead of the current render-in-advance stream path.
    C64 SFX would additionally need a SID synthesizer.
